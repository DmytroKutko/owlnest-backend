# Spring Orchestrated Feature

Use for non-trivial features or refactors that cross a controller, service, domain, persistence, Redis, security, integration, or deployment boundary. Apply the bounded tier route; crossing layers does not automatically justify the full specialist catalog.

## Required context

Read `AGENTS.md`, `docs/project-context.md`, `docs/agent-system/workflow-routing.md`, and only the owning feature/risk documents needed for the task. Preserve `MASTER_REQUEST` verbatim and mark Draft material as proposal. Do not load every handoff or convention document by default.

## Classify first

Before delegation record `DELIVERY_TIER` (`FAST`, `STANDARD`, or `EXCEPTIONAL`), active-time target/checkpoint, expected pass range and count, pass target, impacts, mandatory gates, and `NOT_APPLICABLE` gates. One started or resumed subagent turn is one visible pass.

## Bounded flow

1. **Specify:** the root writes concise criteria when behavior is clear. Use analysts only for material ambiguity or complex traceability.
2. **Discover:** the root inspects focused evidence. Use either rules scout or repository explorer when it saves work; use both only for distinct substantial questions.
3. **Design:** invoke only specialists triggered by a concrete risk. Existing JWT protection and routine additive PostgreSQL changes remain eligible for `FAST`.
4. **Implement:** one `spring_backend_developer` context owns production code and may add focused `FAST` tests. Use separate test engineering only for independently complex test work.
5. **Review:** one independent reviewer plus one focused verification lane is normally enough. New modules retain the skeleton gate; exceptional/high-risk changes retain specialist and release gates.
6. **Correct:** fix only blocking findings—those that invalidate criteria, correctness, security/data safety, required architecture, compilation, migration, or required tests. One targeted correction cycle is normal; a second is only for an unresolved introduced blocking Critical/High. Reuse the same reviewer context only when targeted independent revalidation is needed.
7. **Finish:** when criteria and required evidence pass and no blocking finding remains, stop. Record non-blocking Medium/Low improvements without another writer, reviewer, QA, suite, or release cycle. Reuse observed evidence while code is unchanged and run the broadest justified suite once after the final blocking correction.

Every assignment is one bounded pass with exact scope, supplied evidence, a target shorter than the 15-minute stalled-job guardrail, and concise output. At the tier checkpoint start no optional work; continue only the first or second targeted blocking correction and report why. Missing mandatory evidence remains honest `NOT_RUN`/blocked status, but time never authorizes an unsafe release or an unnecessary polish iteration.
